# 收银系统交付改进计划

本文档的目标不是列想法，而是给这个项目一条从“当前原型”走到“可交付给便利店、多店使用的软件”的实施路线。默认交付形态为：

- 前端：Electron 桌面收银客户端
- 后端：Node.js API 服务
- 数据库：MySQL
- 使用对象：总部管理员、门店管理员、收银员
- 适用场景：单店、多店、断网可收银、联网可同步、可安装部署

---

## 一、先明确交付标准

只有同时满足下面几类要求，这个项目才算“可以交付”。

### 1. 业务可用

- 支持管理员和员工分权
- 支持多门店数据隔离
- 支持商品、库存、销售、退款、报表、员工管理
- 支持交接班、日结、审计日志
- 支持小票打印、扫码枪
- 支持断网收银与恢复同步

### 2. 软件可安装

- 能打包成 Windows 安装包
- 客户机器无需手动安装 Node、npm、数据库客户端
- 安装、升级、配置流程明确
- 本地运行和云端运行两种模式至少确定一种正式方案

### 3. 数据可靠

- 订单、库存、退款不能靠前端本地缓存做唯一数据源
- 有数据库备份、恢复、迁移方案
- 多台设备下数据一致
- 离线订单补传后不会重复记账

### 4. 安全合格

- 密码不能明文存储
- 权限不能只靠前端隐藏按钮
- 重要操作有审计日志
- 接口、配置、密钥、安装包都有基本安全控制

### 5. 可维护

- 有错误日志
- 有基本测试
- 有初始化脚本、部署文档、运维文档
- 有版本升级策略

---

## 二、当前项目现状判断

基于现有代码，当前项目已经有一些可用基础，但距离正式交付还有明显缺口。

### 已有基础

- 已经使用 Electron 打包桌面端
- 已经有 Node.js + Express 后端
- 已经接入 MySQL
- 已经出现 `merchant_id`、`store_id`、`role` 等多租户基础字段
- 前端已有登录、门店切换、报表、退款、审计日志等雏形

### 当前阻塞问题

#### 1. 本地缓存仍然是核心数据源

- `app.js` 中大量使用 `localStorage` 保存商品、交易、统计
- 这适合演示，不适合多门店、多设备正式运行

#### 2. 认证与权限不达标

- 密码仍为明文比对
- JWT secret 存在默认值
- 多数接口只有“登录校验”，没有真正的角色授权和门店授权
- 前端有些管理接口请求甚至没有带鉴权头

#### 3. 多门店能力不完整

- 有门店切换 UI，但后端没有把“管理员可跨店、员工仅本店”的规则完全落实
- 报表部分仍然读取本地交易数据，不是后端真实汇总

#### 4. 桌面软件交付方式不稳

- Electron 通过 `spawn npm` 启动后端
- 这依赖客户机器环境，不适合正式安装包

#### 5. 业务闭环不完整

- 没有完整的交接班、日结、盘点、调拨、采购入库
- 退款流程过于简化
- 没有正式支付状态流转

#### 6. 缺少测试与运维能力

- 缺少自动化测试
- 缺少备份恢复、升级脚本、初始化脚本
- 缺少正式部署说明

## 三、建议的正式产品模型

为避免后面越改越乱，先确定目标架构。

### 1. 角色模型

建议至少采用三级角色：

- `merchant_admin`
  - 总部管理员
  - 管商户、所有门店、所有员工、全局报表、系统配置
- `store_manager`
  - 门店管理员
  - 管本店员工、本店商品、本店库存、本店退款审批、本店报表
- `cashier`
  - 收银员
  - 仅能登录本店、开单、收款、查看自己的班次和订单、发起退款申请

如果你想先快一点，也可以先做两级：

- `admin`
- `cashier`

但数据库和接口最好直接按三级角色预留，否则后续还要返工。

### 2. 数据范围规则

- 总部管理员：可看本商户全部门店
- 门店管理员：只能看自己 `store_id`
- 收银员：只能操作自己 `store_id`，且只允许收银相关功能

### 3. 数据归属原则

所有核心业务数据必须有明确归属：

- 商品：`merchant_id + store_id`
- 员工：`merchant_id + store_id`
- 订单：`merchant_id + store_id + terminal_id + operator_id`
- 审计日志：`merchant_id + store_id + user_id`
- 班次：`store_id + cashier_id`

### 4. 正式部署方案建议

推荐优先采用下面的方案：

- 总部或云端部署一个中心 API + MySQL
- 每家门店安装 Electron 客户端
- 客户端联网时实时请求 API
- 客户端离线时写入本地离线队列，恢复联网后补传

不建议长期采用“每台电脑自带一个本地 Express 服务 + 本地数据库”的模式，运维成本会明显更高。

## 四、总体实施顺序

整个项目建议分 8 个阶段推进：

1. P0：基线收敛与技术债止血
2. P1：认证、权限、门店隔离重构
3. P2：订单与库存核心闭环重构
4. P3：员工管理、班次、报表、审计完善
5. P4：硬件与支付集成
6. P5：离线能力与同步可靠性建设
7. P6：桌面软件打包、安装、升级、部署
8. P7：测试、试点、验收、上线

下面按阶段展开。

## 五、P0：基线收敛与技术债止血

### 目标

先把当前项目中最危险的设计收住，避免后续在错误基础上继续开发。

### 必做事项

#### 1. 固定正式技术路线

- 确定正式使用 MySQL 作为唯一主数据源
- 明确 `localStorage` 只保留为缓存，不再作为订单和库存主来源
- 明确交付形态是“Electron 客户端 + 远端 API”

#### 2. 清理当前代码中的过渡逻辑

- 梳理 `app.js` 中本地数据与云端数据的边界
- 标记所有依赖 `loadTransactions()`、`saveTransactions()`、`loadProducts()`、`saveProducts()` 的核心流程
- 停止继续往 `localStorage` 方案上叠新功能

#### 3. 补齐开发规范

- 建立 `.env.example` 的正式版本
- 增加开发、测试、生产环境说明
- 补充数据库初始化脚本和迁移脚本使用说明

### 涉及文件

- `app.js`
- `server/db.js`
- `server/index.js`
- `README.md`
- `server/mysql_schema.sql`

### 交付结果

- 形成统一开发基线
- 所有人知道后续功能都要基于后端主数据源开发

### 验收标准

- 不再新增以 `localStorage` 为主的核心业务逻辑
- 有一份清晰的环境启动和数据库初始化文档

## 六、P1：认证、权限、门店隔离重构

### 目标

把“管理员和员工分开”“多门店隔离”真正做到后端，而不是只体现在前端界面。

### 必做事项

#### 1. 重构用户表与角色模型

- 将 `users.role` 正式扩展为：
  - `merchant_admin`
  - `store_manager`
  - `cashier`
- 每个员工必须绑定 `store_id`
- 增加字段：
  - `status`
  - `last_login_at`
  - `must_change_password`
  - `created_by`

#### 2. 密码安全改造

- 接入 `bcrypt`
- 所有用户密码改为哈希存储
- 删除默认明文账号方案
- 新建账号后要求首次改密

#### 3. JWT 与会话安全

- 删除默认 JWT secret
- token 中带上：
  - `user_id`
  - `merchant_id`
  - `store_id`
  - `role`
- 增加 token 过期处理和刷新策略

#### 4. 建立统一 RBAC 中间件

新增中间件，例如：

- `authenticate`
- `requireRole([...])`
- `requireStoreScope()`

核心规则：

- 收银员不能请求其他门店数据
- 门店管理员不能请求总部级数据
- 只有管理员能增删员工、调价、删除商品、跨店查看数据

#### 5. 重构认证接口

规范接口：

- `POST /auth/login`
- `POST /auth/logout`
- `GET /auth/me`
- `POST /auth/change-password`
- `POST /users`
- `PATCH /users/:id/status`
- `PATCH /users/:id/reset-password`

删除或重构当前过渡接口：

- `/register-subordinate`
- `/link-subordinate`

#### 6. 前端按权限重组界面

将菜单和页面权限拆清楚：

- 收银员：
  - 收银台
  - 我的订单
  - 我的班次
- 门店管理员：
  - 本店订单
  - 商品
  - 库存
  - 员工
  - 本店报表
- 总部管理员：
  - 全店汇总
  - 门店管理
  - 全局员工
  - 系统设置

### 涉及文件

- `server/routes/auth.js`
- `server/middleware/auth.js`
- `server/db.js`
- `server/mysql_schema.sql`
- `app.js`
- `index.html`

### 交付结果

- 管理员与员工真正分权
- 多门店访问边界由后端保证

### 验收标准

- 收银员不能查看或操作其他门店数据
- 管理端接口都需要正确鉴权
- 密码数据库中不再出现明文

## 七、P2：订单与库存核心闭环重构

### 目标

把收银系统最核心的“下单、扣库存、退款、取消、日结统计”从前端拼装逻辑改成后端事务逻辑。

### 必做事项

#### 1. 重构订单模型

建议新增或完善订单字段：

- `order_no`
- `status`
  - `pending`
  - `paid`
  - `cancelled`
  - `refund_requested`
  - `refunded`
- `payment_status`
- `payment_method`
- `store_id`
- `terminal_id`
- `cashier_id`
- `refund_reason`
- `refunded_at`

#### 2. 用后端事务处理下单

支付成功时，由后端原子完成：

1. 校验商品与库存
2. 创建订单
3. 写入订单明细
4. 扣减库存
5. 写入库存流水
6. 写入审计日志

不能再由前端先扣库存、再异步同步。

#### 3. 建立库存流水机制

新增库存流水表，例如：

- `inventory_movements`

记录类型包括：

- 销售扣减
- 退款回补
- 采购入库
- 盘点调整
- 调拨出库
- 调拨入库
- 损耗报损

#### 4. 完善退款流程

不要只做“直接把原单标记退款”。

应改为：

- 收银员可发起退款申请
- 门店管理员可审批退款
- 审批通过后生成退款记录
- 自动回补库存
- 记录原单、退款单、操作人、原因

#### 5. 商品与库存拆层

将“商品档案”和“门店库存”拆开：

- 商品主档：商户级
- 门店库存：门店级

这样才能支持：

- 总部维护商品资料
- 门店有独立库存
- 后续做调拨和盘点

### 涉及文件

- `server/mysql_schema.sql`
- `server/db.js`
- `server/routes/transactions.js`
- `server/routes/products.js`
- `app.js`

### 交付结果

- 订单、库存、退款形成闭环
- 财务与库存统计不再依赖本地缓存拼凑

### 验收标准

- 一笔订单不会出现“扣库存成功但订单失败”或反过来的情况
- 退款后库存正确回补
- 多台设备同时收银时库存不会明显错乱

## 八、P3：员工管理、班次、报表、审计完善

### 目标

补齐便利店日常运营所需的后台管理能力。

### 必做事项

#### 1. 员工管理

管理员端支持：

- 新增员工
- 停用员工
- 重置密码
- 绑定门店
- 查看最近登录时间

#### 2. 班次管理

新增班次流程：

- 上班签到
- 下班交班
- 本班次现金统计
- 班次销售汇总
- 班次差异说明

建议新增表：

- `shifts`
- `shift_transactions`

#### 3. 日结与报表

管理员报表分层：

- 门店日报
- 门店月报
- 商品销售排行
- 类目销售排行
- 员工业绩
- 退款统计
- 毛利报表（如果后面录入进货成本）

注意：

- 所有报表都以后端数据库聚合生成
- 不再读取前端本地交易缓存

#### 4. 审计日志完善

审计日志要覆盖：

- 登录
- 改价
- 删除商品
- 退款
- 盘点调整
- 员工创建/停用
- 门店切换

审计日志必须记录：

- 谁做的
- 在哪个门店做的
- 做了什么
- 改前改后是什么
- 时间

### 涉及文件

- `server/routes/audit.js`
- `server/routes/auth.js`
- `server/routes/transactions.js`
- `server/db.js`
- `app.js`
- `index.html`

### 交付结果

- 店长和总部可以真正管理门店运营
- 出问题时可追溯

### 验收标准

- 管理员能按门店、按日期查看销售数据
- 员工操作都有审计记录
- 交接班过程完整可查

## 九、P4：硬件与支付集成

### 目标

让软件从“网页收银”变成“门店能用的收银软件”。

### 必做事项

#### 1. 小票打印

支持热敏打印机：

- 58mm / 80mm 小票
- 打印内容可配置
- 支持补打小票
- 支持打印退款单

实现方式建议：

- Electron 主进程调用本地打印能力
- 或接入成熟打印组件

#### 2. 扫码枪支持

优化扫码录入体验：

- 自动聚焦
- 连续扫码不卡顿
- 识别商品码、支付码
- 异常码提示明确

#### 3. 钱箱与客显

如果客户门店需要：

- 结账后自动弹钱箱
- 客显屏展示金额

这部分可作为可选能力。

#### 4. 支付状态流

即使暂时不直连官方支付，也要先把流程设计正确：

- 待支付
- 支付中
- 已支付
- 支付失败
- 已退款

后续若接支付接口，可平滑接入。

### 涉及模块

- `main.js`
- `preload.js`
- `app.js`
- 新增 `printer`、`device`、`payment` 相关模块

### 交付结果

- 门店能实际完成收银与出小票

### 验收标准

- 完成支付后可自动打印小票
- 连续扫码稳定
- 支持补打上一单小票

## 十、P5：离线能力与同步可靠性建设

### 目标

让门店断网时还能卖货，恢复网络后数据不乱。

### 必做事项

#### 1. 重做离线模式

当前“点击按钮直接以离线收银员进入系统”的方式不能用于正式环境。

正式方案应为：

- 只有本机已登录过且已授权的账号才能离线登录
- 离线状态下只允许当前门店、当前账号操作
- 离线订单写入本地队列，不直接当作正式云端主记录

#### 2. 建立本地离线队列

建议使用：

- IndexedDB 或 SQLite

记录内容：

- 待同步订单
- 待同步退款
- 待同步审计日志
- 待同步库存调整

#### 3. 同步策略

为每条业务数据设计：

- 本地唯一 id
- 云端幂等 key
- 同步状态
- 重试次数
- 冲突处理规则

#### 4. 冲突处理

必须定义清楚：

- 同一商品离线期间被多台机卖出怎么办
- 离线退款恢复后如何审核
- 重复同步如何避免重复记账

### 涉及模块

- `app.js`
- 新增本地离线存储模块
- `server/routes/transactions.js`
- `server/routes/audit.js`

### 交付结果

- 断网不中断收银
- 恢复后数据一致

### 验收标准

- 断网 1 小时可持续收银
- 恢复网络后订单自动补传
- 不出现重复订单和重复扣库存

## 十一、P6：桌面软件打包、安装、升级、部署

### 目标

把项目变成真正能交给客户安装的软件，而不是开发环境项目。

### 必做事项

#### 1. 重构桌面端启动方式

不要再依赖 `spawn npm` 启动后端。

建议二选一：

方案 A，推荐：

- Electron 客户端只负责前端和设备能力
- 后端部署在云端或总部服务器

方案 B：

- 把本地服务打包成独立后台程序
- 安装时自动注册服务

#### 2. 安装包与配置

安装包至少支持：

- Windows 安装版
- 桌面快捷方式
- 首次启动配置服务器地址
- 设备编号配置

#### 3. 自动更新

至少支持：

- 手动检查更新
- 版本号展示
- 升级记录

后续可加自动更新。

#### 4. 初始化工具

提供交付工具：

- 初始化商户
- 初始化门店
- 初始化管理员
- 导入基础商品

### 涉及文件

- `package.json`
- `main.js`
- `preload.js`
- 新增部署文档、初始化脚本

### 交付结果

- 客户机器可直接安装并运行

### 验收标准

- 新电脑安装后 30 分钟内可完成首店初始化并开始收银
- 无需额外安装 Node/npm 才能运行

## 十二、P7：测试、试点、验收、上线

### 目标

把“开发完成”变成“可上线交付”。

## 二十、最终目标

这个项目的最终目标，不是做一个“好看的收银页面”，而是做一个门店敢拿来营业的软件。

判断标准只有一个：

- 店员能用它稳定收银
- 店长能用它管理门店
- 老板能用它看清经营数据
- 出问题时你能查、能修、能恢复

只要按本文档的顺序推进，这个项目会从原型逐步收敛成一个可以真正交付的便利店收银系统。

## 二十一、第一阶段开发任务清单

第一阶段只解决最核心的问题：

- 管理员和员工真正分权
- 多门店数据真正隔离
- 登录和密码达到基本安全要求
- 前端页面权限与后端权限一致

这一阶段先不重做订单事务、不接支付、不做打印。目标是先把系统“边界”做对。

### 1. 阶段目标

完成后，系统必须达到下面状态：

- 总部管理员可以查看和管理本商户下所有门店
- 门店管理员只能管理自己门店
- 收银员只能在自己门店登录和收银
- 所有接口都按 `merchant_id + store_id + role` 做后端校验
- 密码使用哈希存储，不再使用明文
- 前端菜单只是“显示优化”，不是权限本体

### 2. 第一阶段范围

#### 本阶段必做

- 用户表结构升级
- 密码哈希
- JWT 载荷规范化
- RBAC 权限中间件
- 门店范围校验
- 登录接口重构
- 当前用户信息接口
- 员工管理基础接口
- 门店列表接口权限收敛
- 前端登录、角色菜单、门店切换重构

#### 本阶段不做

- 订单事务化
- 退款审批流重构
- 采购、盘点、调拨
- 小票打印
- 支付接口
- 离线队列重构

## 二十二、第一阶段数据库改造

### 1. users 表调整

当前 `users` 表建议扩充为以下字段：

- `id`
- `merchant_id`
- `store_id`
- `username`
- `password_hash`
- `name`
- `role`
- `status`
- `must_change_password`
- `last_login_at`
- `created_by`
- `created_at`
- `updated_at`

### 2. 角色取值

`role` 建议改为：

- `merchant_admin`
- `store_manager`
- `cashier`

如果你想兼容当前前端，也可以先临时保留：

- `admin`
- `cashier`

但迁移脚本里要明确未来映射：

- `admin -> merchant_admin`

### 3. 状态字段

`status` 建议支持：

- `active`
- `disabled`
- `locked`

### 4. SQL 改造任务

在 `server/mysql_schema.sql` 中完成：

1. 为 `users` 新增 `password_hash`
2. 为 `users` 新增 `status`
3. 为 `users` 新增 `must_change_password`
4. 为 `users` 新增 `last_login_at`
5. 为 `users` 新增 `created_by`
6. 修改 `role` 的枚举值
7. 逐步废弃旧 `password` 字段

### 5. 迁移策略

建议按两步迁移，不要一步硬切：

#### 第一步

- 保留旧 `password`
- 新增 `password_hash`
- 启动后检测没有 `password_hash` 的用户并迁移

#### 第二步

- 所有账号完成迁移后
- 删除旧 `password`

这样不会把现有测试账号和历史数据一次性搞坏。

## 二十三、第一阶段后端文件改造清单

### 1. `server/mysql_schema.sql`

任务：

- 调整 `users` 表结构
- 补充默认门店管理员、收银员测试数据
- 默认测试数据改为哈希密码

输出结果：

- 一个可直接初始化的正式基础表结构

### 2. `server/db.js`

任务：

- `getUserByUsername(username, merchantId)` 改为返回 `password_hash`
- 新增 `getUserById(userId, merchantId)`
- 新增 `createUser(payload, merchantId)`
- 新增 `updateUserStatus(userId, status, merchantId)`
- 新增 `updateUserPassword(userId, passwordHash, merchantId)`
- 新增 `getUsersByStore(storeId, merchantId)`
- 新增 `getStoreById(storeId, merchantId)`

注意：

- 所有查询都必须带 `merchant_id`
- 不允许写出“只按 user id 查全局”的函数

### 3. `server/middleware/auth.js`

任务：

- 删除默认 `JWT_SECRET`
- 新增环境变量缺失时报错逻辑
- JWT payload 统一为：
  - `id`
  - `merchant_id`
  - `store_id`
  - `role`
  - `username`
- `authenticate` 中间件只负责验 token，不负责授权

### 4. 新增 `server/middleware/rbac.js`

建议新增这个文件，集中处理权限。

至少提供 3 个中间件：

- `requireRole(roles)`
- `requireActiveUser`
- `requireStoreScope`

其中：

- `requireRole(roles)`：检查角色
- `requireActiveUser`：检查用户状态是否为 `active`
- `requireStoreScope`：限制是否可访问目标 `store_id`

### 5. `server/routes/auth.js`

任务：

- 重写登录接口
- 登录时使用 `bcrypt.compare`
- 成功登录后更新 `last_login_at`
- 新增 `/me`
- 新增 `/change-password`
- 删除或下线：
  - `/register-subordinate`
  - `/link-subordinate`

### 6. 新增 `server/routes/users.js`

建议新增员工管理路由，不要继续把员工逻辑挤在 `auth.js`。

基础接口：

- `GET /users`
- `POST /users`
- `PATCH /users/:id/status`
- `PATCH /users/:id/password-reset`

### 7. `server/routes/stores.js`

任务：

- 仅返回当前商户可见门店
- 如果是 `store_manager` 或 `cashier`，只返回自己的门店
- 如果是 `merchant_admin`，返回全部门店

### 8. `server/index.js`

任务：

- 注册 `users` 路由
- 注册统一错误处理
- 明确启动时校验环境变量

## 二十四、第一阶段接口设计

### 1. 登录

`POST /api/v1/auth/login`

请求体：

```json
{
  "merchantId": "m_default",
  "username": "admin",
  "password": "123456"
}
```

成功返回：

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "id": "u1",
    "merchant_id": "m_default",
    "store_id": "s1",
    "username": "admin",
    "name": "系统管理员",
    "role": "merchant_admin",
    "must_change_password": false,
    "token": "jwt-token"
  }
}
```

### 2. 获取当前用户

`GET /api/v1/auth/me`

用途：

- 前端刷新页面后恢复用户上下文
- 避免前端只相信本地缓存

### 3. 修改本人密码

`POST /api/v1/auth/change-password`

请求体：

```json
{
  "oldPassword": "old123",
  "newPassword": "new123456"
}
```

### 4. 获取员工列表

`GET /api/v1/users?store_id=s1`

规则：

- `merchant_admin` 可看全部或指定门店
- `store_manager` 只能看自己的 `store_id`
- `cashier` 禁止访问

### 5. 创建员工

`POST /api/v1/users`

请求体：

```json
{
  "username": "cashier01",
  "name": "张三",
  "password": "123456",
  "role": "cashier",
  "store_id": "s1"
}
```

规则：

- `merchant_admin` 可创建任意门店员工和店长
- `store_manager` 只能创建本店 `cashier`

### 6. 停用员工

`PATCH /api/v1/users/:id/status`

请求体：

```json
{
  "status": "disabled"
}
```

### 7. 重置密码

`PATCH /api/v1/users/:id/password-reset`

请求体：

```json
{
  "newPassword": "123456"
}
```

### 8. 获取可见门店

`GET /api/v1/stores`

规则：

- 总部管理员返回全部门店
- 门店管理员与收银员仅返回自己的门店

## 二十七、第一阶段验收用例

- `cashier@s1` 无法看到 `s2` 商品和订单
- `store_manager@s1` 无法查看 `s2` 员工
- `merchant_admin` 可以查看全部门店

### 4. 前端显示

- 收银员看不到门店切换器
- 收银员看不到商品管理敏感功能
- 管理员可以看到员工管理页
