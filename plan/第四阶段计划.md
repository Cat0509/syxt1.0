# 第四阶段计划（基础重构期）

更新日期：2026-04-07

定位说明：

- 第四阶段对应 [后续整体开发计划.md](/d:/1rjkf/syxt/plan/后续整体开发计划.md) 中的“基础重构期”
- 它不是把后续所有高优先级事项一次做完，而是先处理最容易导致返工的数据模型、技术债务和离线收银底座
- 第四阶段完成后，第五阶段再进入“业务闭环期”，补 `refund_items`、`order_payments`、`store_prices` 等完整业务能力
- 当前状态：已于 2026-04-07 通过封版验收，详见 [第四阶段封版验收记录.md](/d:/1rjkf/syxt/plan/第四阶段封版验收记录.md)

## 一、阶段目标

在第三阶段“数据后端化收口、支付 Mock 闭环、离线回放幂等”基础上，第四阶段要把系统从“功能可用”推进到“架构可持续演进”。

本阶段核心目标是：

- 冻结后续演进所需的数据模型底座
- 清理当前最危险的技术债务
- 完成正式离线收银能力的底层重构
- 为第五阶段的部分退款、混合支付、门店差异价格预留稳定结构
- 建立可持续的迁移、日志和测试基础

本阶段不追求把所有业务功能一次做完，重点是先把“边界”和“地基”做对。

---

## 二、第四阶段交付标准

- `products` 与 `inventory` 完成职责拆分，不再双写库存字段
- 主键与业务单号生成策略统一，不再使用 `Date.now() + random`
- 新增订单不再依赖 `transactions.items` JSON 作为主明细来源
- 正式离线收银底座可运行：
  - 设备绑定
  - 本机离线登录
  - 本地挂单队列
  - 价格快照
  - 营业日字段
  - 挂单失效释放库存
- 数据库进入版本化迁移模式
- 核心链路具备基础结构化日志和 smoke test

---

## 三、本阶段不做

以下内容明确延后到后续阶段，避免第四阶段范围失控：

- `refund_items` 驱动的按商品部分退款完整闭环
- `order_payments` 驱动的在线混合支付正式闭环
- `store_prices` 驱动的门店差异价格正式业务闭环
- 真实微信、支付宝接入
- Electron 与云端正式分离交付
- 班次交接、盘点任务、库存预警页面

第四阶段只做这些能力的“结构预留”和“底层准备”，不做完整业务交付。

## 四、核心范围

### 4.1 注册与初始化流程重做

目标：

- 把当前“公开注册开店”重构为正式初始化流程

本阶段要做：

- 明确“商户 / 总部管理员 / 首店 / 首台设备”初始化顺序
- 首次初始化只允许创建第一位总部管理员
- 初始化完成后才进入正常登录逻辑
- 为后续多门店扩展保留结构

验收结果：

- 可以完成一次从空库到首商户、首管理员、首店、首设备的初始化

### 4.2 商品主档与库存正式拆层

目标：

- 彻底终止当前商品表兼管库存的状态

本阶段要做：

- 停止把 `products.stock` 作为正式库存来源
- 停止把 `products.store_id` 作为正式门店维度来源
- 停止 `skus.stock` / `skus.store_id` 的沿用
- 所有库存统一回归 `inventory`
- 清理 `db.js`、库存接口和商品同步中的双写逻辑
- 给第五阶段的 `store_prices` 留出干净结构

验收结果：

- 任意库存变化只写 `inventory`
- 商品主档与库存数据不再相互污染

### 4.3 ID 生成策略统一

目标：

- 消除现有拼接式 ID 的碰撞风险

本阶段要做：

- 定义统一 ID 工具
- 区分“内部主键”和“对外展示单号”
- 替换订单、退款、支付、离线挂单等关键对象的 ID 生成逻辑
- 补迁移兼容策略，避免新旧 ID 混用失控

建议策略：

- 内部主键：UUID v4
- 展示单号：独立业务编号规则

验收结果：

- 核心新增记录不再使用 `Date.now() + random`

### 4.4 废弃 `transactions.items` JSON

目标：

- 明确订单明细的唯一事实来源

本阶段要做：

- 停止新增写入 `transactions.items`
- 查询订单明细统一从 `order_items` 读取
- 兼容历史数据读取
- 设计 `transactions.items` 最终删除节点

验收结果：

- 新订单明细查询不再依赖 JSON 列

### 4.5 离线收银底座重构

目标：

- 把第三阶段“离线回放”升级成可支撑正式演进的底层能力

本阶段要做：

- 本地离线队列从简单 `pending_orders` 升级为正式挂单存储
- 绑定 `device_id`
- 只允许本机登录过的账号离线登录
- 记录 `price_snapshot`
- 记录 `business_date`
- 支持 `offline_cash_collected`
- 支持 `offline_payment_pending`
- 支持未支付挂单跨营业日失效归档并释放库存
- 为后续“恢复联网后处理挂单”预留状态流转

本阶段重点是“数据模型和本地状态机”，不是把所有前台交互细节做到最终版。

验收结果：

- 应用重启后挂单仍可恢复
- 离线登录只允许本机已登录过的账号
- 未支付挂单可在营业日结束后失效归档

### 4.6 数据库版本化迁移

目标：

- 结束一次性 schema 脚本模式

本阶段要做：

- 建立版本化迁移目录和编号规范
- 为第四阶段所有 schema 变更补迁移脚本
- 补回滚策略
- 约束后续阶段不再直接手改总 schema 文件作为唯一变更方式

验收结果：

- 第四阶段变更都能通过迁移脚本落地
- 迁移可重复执行，可回滚

### 4.7 结构化日志与基础测试底座

目标：

- 为后续大改提供最低限度可观测性和回归保护

本阶段要做：

- 引入基础结构化日志方案
- 收敛关键链路上的 `console.log`
- 先覆盖这些 smoke test：
  - 初始化流程
  - 离线登录边界
  - 挂单持久化
  - 库存拆层后下单链路
  - 迁移脚本执行

验收结果：

- 第四阶段关键链路具备可追踪日志
- 最低限度 smoke test 可复跑

## 五、数据模型与结构预留

虽然第四阶段不交付完整业务闭环，但必须先把后续表结构和状态流考虑进去。

### 5.1 本阶段必须直接落地

- `device_id`
- `business_date`
- `price_snapshot`
- 离线挂单本地状态字段
- 迁移版本号表或等价机制

### 5.2 本阶段只做结构预留，不做完整业务闭环

- `refund_items`
- `order_payments`
- `store_prices`
- `partially_refunded`

第四阶段文档和迁移设计里要给这些后续表和状态留好接口，不允许再出现“第五阶段一做就推翻第四阶段”的情况。

---

## 六、实施顺序建议

1. 冻结第四阶段数据模型和迁移规范
2. 先做商品主档/库存拆层
3. 再统一 ID 生成策略
4. 再切换 `transactions.items` 读取口径
5. 再做初始化流程和设备绑定
6. 再做离线队列重构
7. 最后补日志、测试和收口

这样排的原因是：

- 先动数据底层，避免后面离线逻辑又绑到旧结构上
- 先统一 ID，避免本地挂单和后端订单再次产生兼容烂尾
- 先收掉 JSON 明细，再做离线状态机，后续查询口径才不会混乱

## 九、风险与注意事项

- 商品主档与库存拆层是本阶段最大风险点，改动会波及下单、库存、商品同步和报表
- `transactions.items` 废弃必须保留历史兼容读取，不能直接硬删
- ID 策略切换期间要避免新旧格式混用导致查询和排序异常
- 离线收银底座重构不能直接跳到完整 UI，先把本地状态机做稳
- 迁移体系一旦开头做歪，后续每个阶段都会被拖累，所以本阶段必须先定规范
