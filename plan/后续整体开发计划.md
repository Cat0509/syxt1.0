## 1. 总体目标

后续开发不再按“单个阶段补丁式推进”，而是按一条完整产品路线推进。

系统的整体目标是：

- 服务当前 3 家自营门店
- 以中心化部署支撑门店收银和管理员在线查看
- 管理员看全局，店长只看本店
- 中心服务断网时，门店仍可继续收银
- 后续可平滑接入真实微信和支付宝支付
- 保留未来扩展到更多商户的结构可能

## 2. 当前阶段盘点

### 2.1 已完成阶段

- 第一阶段：权限与门店隔离已封版
- 第二阶段：订单事务化与库存流水已落地
- 第三阶段：数据后端化收口、报表后端化、支付 Mock 闭环、离线回放幂等已完成
- 第四阶段：基础重构期已于 2026-04-07 验收通过

### 2.2 当前遗留项

- 真实第三方支付接入仍是外部依赖项
- 当前正式进入第五阶段“业务闭环期”准备阶段

### 2.3 这份计划的定位

这份文档不是“第四阶段计划”，而是从现在开始到正式上线及后续增强的整体路线图。

## 3. 已锁定的产品规则

当前整体规则已经定稿如下：

- 离线收银采用 `K2`
  - 断网时允许挂单
  - 扫码支付必须等网络恢复后再完成
  - 现金可以离线先收款，恢复后再同步
- 挂单占用预留库存
- 挂单记录至少保留 3 年
- 未支付挂单在当前营业日结束后失效归档并释放库存
- 恢复联网后按挂单价格快照成交
- 只有本机登录过的账号才能离线登录，且设备绑定 `device_id`
- 日汇总按营业日统计，默认切点为 `06:00`
- 混合支付退款按“原支付渠道优先退回，超出部分现金补退”
- 电子小票优先，纸质小票按需打印

## 4. 本次补齐的 16 项缺口

### 4.1 高优先级

这些内容必须并入基础重构期和业务闭环期。

#### 数据模型缺失

1. 部分退款
- 缺 `refund_items` 表
- 缺 `partially_refunded` 状态

2. 混合支付
- 缺 `order_payments` 表
- 当前 `payment_method` 仍是单选口径

3. 门店差异价格
- 缺 `store_prices` 表

#### 技术债务

4. 商品表拆层
- `products.stock` / `products.store_id` 已标 Deprecated，但代码仍在双写

5. ID 生成策略
- 当前主要用 `Date.now() + random` 拼接，存在碰撞风险

6. `transactions.items` JSON 列
- 与 `order_items` 表重复，需要明确废弃节点

### 4.2 中优先级

这些内容需要并入基础重构期、上线交付期和横跨阶段治理。

7. Electron 打包与后端正式分离方案
8. HTTPS 和 CORS 生产配置

10. 断网冲突的具体处理规则
11. 结构化日志方案
12. 自动化测试策略
13. 数据库版本化迁移方案

### 4.3 低优先级

这些内容归入运营增强期。

14. 班次交接

## 5. 整体推进顺序

后续开发按 4 个时期推进：

1. 基础重构期
2. 业务闭环期
3. 上线交付期
4. 运营增强期

下面的各时期已经把上面的缺口正式并入。

## 6. 基础重构期

### 6.1 目标

- 把系统从“能跑”改成“可长期演进”
- 先处理最容易导致后续返工的数据模型和基础架构问题

### 6.2 核心内容

#### 6.2.1 注册和初始化流程重做

从“公开注册开店”改成正式的商户初始化或开通流程，明确：

- 商户
- 总部管理员
- 首店
- 默认设备与账号初始化

#### 6.2.2 权限与门店范围收口

明确：

- 管理员可看全局
- 店长只看本店
- 收银员只操作本终端和本店

#### 6.2.3 商品主档与门店库存正式拆层

这是基础重构期的硬性项。

必须落地：

- 正式废弃 `products.stock`
- 正式废弃 `products.store_id`
- 正式废弃 `skus.stock`
- 正式废弃 `skus.store_id`
- `products` 回归商户级商品主档
- 所有库存统一以 `inventory` 为准
- 删除现有双写逻辑

#### 6.2.4 统一 ID 生成策略

必须落地：

- 不再使用 `Date.now() + random`
- 统一采用稳定主键策略
- 推荐优先级：
  - 服务端内部主键：UUID v4 或数据库 UUID
  - 对外展示单号：单独的业务单号生成策略

#### 6.2.5 废弃 `transactions.items` JSON

必须落地：

- 停止新增写入 `transactions.items`
- 订单明细统一从 `order_items` 读取
- 明确迁移窗口和最终删除节点

#### 6.2.6 离线收银能力升级

围绕已定规则落地：

- 现金离线完成待同步
- 扫码离线挂单待联网支付
- 挂单占库存
- 失效归档释放库存
- 价格快照成交
- `device_id` 绑定
- 营业日字段

#### 6.2.7 数据库版本化迁移方案

必须补齐：

- 每次 schema 变更都要有版本号
- 每次变更都要有迁移脚本
- 每次迁移都要有回滚方案

### 6.3 交付物

- 初始化流程设计和接口说明
- 商品主档/库存拆层迁移方案
- ID 生成策略说明
- `transactions.items` 废弃计划
- 版本化迁移规范与首批迁移脚本
- 离线收银升级设计说明

### 6.4 验收标准

- 产品主档与库存不再双写
- 新增订单不再依赖 `transactions.items` JSON
- 主键生成策略统一
- 离线收银状态机可跑通
- 迁移脚本可重复执行且可回滚

## 7. 业务闭环期

### 7.1 目标

- 把门店高频使用的业务流程补完整
- 补齐高优先级缺失的数据模型

### 7.2 核心内容

#### 7.2.1 部分退款数据模型

必须新增：

- `refund_items`

建议字段：

- `refund_id`
- `order_item_id`
- `sku_id`
- `qty`
- `amount`
- `reason`

必须补齐：

- 订单状态新增 `partially_refunded`
- 审批通过后按行项目回补库存
- 部分退款后的报表和对账口径

#### 7.2.2 混合支付数据模型

必须新增：

- `order_payments`

建议字段：

- `id`
- `order_id`
- `method`
- `amount`
- `status`
- `payment_id`
- `meta`

同时处理：

- `transactions.payment_method` 从单一事实来源降级为兼容字段，最终只保留展示用途或废弃
- 支付状态从“整单一个字段”演进为“整单汇总 + 支付明细表”

#### 7.2.3 门店差异价格数据模型

必须新增：

- `store_prices`

建议字段：

- `store_id`
- `product_id`
- `sku_id`
- `price`
- `updated_by`
- `updated_at`

同时处理：

- 收银价格读取优先级
  - `store_prices`
  - `skus.price`
  - `products.price`
- 价格变更审计日志

#### 7.2.4 对账中心

至少做到门店维度的：

- 应收
- 实收
- 退款
- 差异
- 异常订单列表

#### 7.2.5 审计日志

覆盖：

- 登录
- 改价
- 库存调整
- 退款申请和审批
- 订单作废
- 挂单创建、取消、补传、恢复支付
- 配置修改

### 7.3 交付物

- `refund_items` 数据模型与接口说明
- `order_payments` 数据模型与支付流说明
- `store_prices` 数据模型与优先级说明
- 对账中心接口与页面
- 审计日志表结构与查询页面

### 7.4 验收标准

- 支持按商品部分退款
- 支持在线混合支付
- 支持门店差异化价格
- 对账页面能定位差异订单
- 审计日志能按门店、账号、时间查询

## 8. 上线交付期

### 8.1 目标

- 把系统从“业务能用”推到“门店可稳定上线”
- 处理部署、安全、运维和工程规范问题

### 8.2 核心内容

#### 8.2.1 Electron 打包与后端正式分离

正式采用：

- 云端 API + Electron 纯客户端架构

Electron 负责：

- 页面渲染
- 设备能力
  - 打印机
  - 扫码枪
  - 钱箱
- 本地离线存储

服务端负责：

- 认证
- 数据存储
- 支付
- 对账
- 审计
- 多门店管理

同时处理：

- `main.js` 中本地拉起后端的方案要退出正式版
- Electron 打包、更新、版本兼容策略

#### 8.2.2 HTTPS 和 CORS 生产配置

必须补齐：

- 生产 API 统一走 HTTPS
- CORS 白名单化
- JWT 传输与刷新策略
- 环境变量与密钥管理

#### 8.2.3 数据库运维

必须补齐：

- 连接池配置
- 慢查询监控
- 自动备份策略
- 恢复演练
- 数据库容量与索引巡检

#### 8.2.4 断网冲突处理规则

必须写死规则：

- 多台设备离线卖超同一库存时怎么处理
- 商品下架后的挂单如何恢复
- 价格变更后的挂单如何提示
- 自动重试次数与人工介入节点

#### 8.2.5 真实支付与设备联调

必须落地：

- 真实微信接入
- 真实支付宝接入
- 打印机联调
- 扫码枪联调
- 支付回调监控

### 8.3 交付物

- Electron 客户端和云端服务拆分方案
- 生产部署清单
- HTTPS/CORS/密钥管理配置说明
- 数据库运维清单
- 断网冲突处理规则表
- 真实支付接入文档

### 8.4 验收标准

- 客户端不再依赖本地启动后端
- 生产环境 HTTPS 可用
- CORS 受控
- 备份可恢复
- 真实支付可联调
- 设备联调通过

## 10. 横跨所有时期的工程治理

这些不是单一时期的功能项，而是需要从现在开始贯穿后续所有时期。

### 10.1 结构化日志

当前项目中存在大量 `console.log` 和 `console.error`，正式环境不能继续沿用。

必须补齐：

- 统一日志框架
- `info / warn / error` 分级
- 请求链路字段
- 支付、同步、库存、登录等关键事件埋点
- 日志轮转和留存策略

### 10.2 前端代码结构重构

当前 `app.js` 体量已经过大，后续继续叠加功能会明显拖慢维护效率。

必须补齐：

- 按功能域拆分前端代码
- 收银、报表、退款、库存、认证、同步逻辑分层
- Electron 壳层与业务层分离

### 10.3 自动化测试策略

必须补齐：

- 后端 API 单元测试
- 关键收银链路的前端 E2E 测试
- 阶段性 smoke test
- 发布前回归测试清单

### 10.4 数据库版本化迁移

虽然在基础重构期首次落地，但后续所有 schema 变更都必须进入该体系。

### 10.5 风险与依赖清单

需要持续维护：

- 真实支付申请周期
- 打印机兼容性
- 门店网络条件
- 自动更新失败回滚
- 数据库升级窗口

## 11. 关键依赖关系

为了避免返工，后续实施必须遵守下面的依赖顺序：

1. 商品主档与库存拆层

8. 真实支付与设备联调
